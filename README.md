# WSE‑MATD3: Weighted Sum Evolution Enhanced Multi‑Agent MARL for Power System Stability

This repository contains the official implementation of **WSE‑MATD3** (Weighted Sum Evolution enhanced Multi‑Agent Twin Delayed Deep Deterministic Policy Gradient), a multi‑agent reinforcement learning framework for coordinated multi‑objective stability enhancement in multi‑area power systems. The method integrates a **distributed real‑time Prony analysis** into the observation space and dynamically balances four conflicting control objectives (frequency regulation, voltage support, oscillation damping, and control effort) via a lightweight weight evolution strategy. The code also includes baseline implementations of MATD3, MAPPO, and MASAC for comparison.

## Key Features

- **Time‑Frequency Augmented Observation**: Each agent receives a hybrid time‑modal percept that concatenates global PMU measurements (generator speeds, voltages, ACEs, COI states) with local Prony features (dominant damping ratio, oscillation frequency, stability index, and inter‑area coordination flags).
- **Multi‑Objective Reward with Weighted Sum Evolution (WSE)**: A population of weight vectors is evolved via elite preservation, crossover, and mutation. The scalar fitness is the average weighted sum of recent episode performances, enabling automatic trade‑off without manual tuning.
- **Decentralized Prony Analysis**: Each area runs a local Prony analyzer on its own measured signals; a lightweight coordinator exchanges only validity flags and normalised counts, preserving data privacy.
- **Centralised Training with Decentralised Execution (CTDE)**: All agents share a centralised critic during training but act solely on local observations during execution.
- **Extensible Framework**: All algorithms (WSE‑MATD3, MATD3, MAPPO, MASAC) share the same ANDES‑based environment and evaluation pipeline.

## Algorithm Overview

WSE‑MATD3 comprises three main components:

1. **Decentralized Prony Analysis**  
   - Each area maintains a sliding window of locally measured COI frequency.  
   - A Hankel matrix is formed and solved via truncated SVD to extract dominant oscillation modes.  
   - Output: damping ratio `ζ`, oscillation frequency `f`, amplitude `A`, and a stability index `ξ = min(2, ζ/0.15)`.  
   - A coordinator computes inter‑area features `[valid_flag, ξ, r]` (r = number of other areas with valid modes / 3).

2. **Multi‑Objective Reward & WSE**  
   - The instantaneous reward for agent `i` is:  
     `r_i = -w1·J_freq - w2·J_volt + w3·J_damp - w4·J_ctrl`  
     where `J_freq = 2000·|Δω| + 5·|ACE|`, `J_volt = 500·(ΔV)²`, `J_damp = 3·ζ`, and `J_ctrl` penalises action magnitude, step changes, and saturation.  
   - Every 10 episodes, the weight population (size 10, initialised from Dirichlet(1,1,1,1)) is evolved: elite preserved, remaining generated by weighted average crossover + Gaussian mutation (`σ_mut=0.05`).  
   - Fitness = average weighted sum over last 10 episodes.

3. **MATD3 Backbone**  
   - Twin critics to mitigate overestimation.  
   - Delayed policy updates (every 2 critic updates).  
   - Target policy smoothing with clipped noise (`σ_noise=0.1`, clip=0.5).  
   - Polyak averaging for target networks (`τ=0.01`).  
   - Exploration noise annealed from 0.05 to 0.01 over training.

## Environment

The environment is built on **ANDES** (a Python‑based power system simulation tool) and models a modified IEEE 39‑bus, 4‑area, 10‑generator system with:
- 9 synchronous generators (GENROU models) with turbine‑governor systems
- 1 grid‑forming VSG (REGF2 model)
- Random disturbances: permanent load increases (severity 0.1–0.4 p.u.) and line trips
- Simulation horizon: 15 s, control actions at 28 discrete time instants (from t=2 s to t=15 s)

The observation space dimension is 50 (global 43 + Prony 7). The action space consists of supplementary power setpoint modulations for each controllable unit, bounded to [-1, 1] p.u.

## Usage
### Repository Structure
```bash
├── andes_marl.py                # Custom multi‑agent environment (MultiAgentAndes39)
├── MARL_train.py                # Main training script (all algorithms)
├── MARL_test.py                 # Testing and visualisation script
├── ieee39_4area_REGF.xlsx       # ANDES case file
├── train/                       # Created during training (saves models, logs)
└── test/                        # Created during testing (results, plots)
```
### Train All Algorithms
 --python MARL_train.py
### Test Trained Models
 --python MARL_test.py
